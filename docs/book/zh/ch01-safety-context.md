# 1. 安全上下文（Safety Context）

每个函数都**默认、无条件**受检查（见 [§5](ch05-static-checks.md)）——不
存在任何按函数、按文件的标注来开启这件事。语言里唯一的安全上下文
构造是 `unsafe { }` 块：一个词法作用域的、局部的、可组合的逃生舱，
只放宽 [§5.5](ch05-static-checks.md) 里那份固定、列举出来的操作清单，
别的什么都不放宽。这是对早先设计（一个 `safe` 关键字，让单个函数
opt-in 检查，没标注的函数完全不检查）的刻意反转——完整理由见
[§8](ch08-open-questions.md) Q13。

## 1.1 `unsafe { }` 放宽了什么，什么保持无条件

- **[§5.1](ch05-static-checks.md)-[§5.4](ch05-static-checks.md) 里的
  所有东西**——所有权/move 状态、别名 XOR 可变、生命周期/悬垂检查、
  零初始化——**无条件、任何地方都跑，包括在 `unsafe { }` 块里面**。
  `unsafe { }` 从来不会关掉这套分析；scpp 里也没有任何构造能做到这
  一点。这跟 Rust 完全一致：Rust 的 `unsafe fn` 函数体、或者
  `unsafe { }` 块内部，借用检查照样完整地跑——`unsafe` 只解锁借用
  检查器自己没法验证的几个具体操作，从来不会把借用检查整个停掉。
- **只有 [§5.5](ch05-static-checks.md) 列出的操作，在 `unsafe { }`
  里才变合法**：裸指针解引用/指针算术、`reinterpret_cast`/不兼容
  类型间的 C 风格强转、未加标签的 `union` 成员访问、裸
  `new`/`delete`、可变全局/静态变量访问、以及调用一个 `extern "C"`
  函数（`extern "C"` 声明永远不会被任何 scpp 编译器检查，见
  [§2.1](ch02-boundary-rules.md)，所以调用它跟清单上其它每一项
  一样，都需要 `unsafe { }` 提供的这份担保）。
- **`span` 的边界检查（[§8](ch08-open-questions.md) Q1）和整数
  溢出检查（[§5.8](ch05-static-checks.md)）在 `unsafe { }` 里也会
  被放宽**，但理由跟上面不一样：不是因为它们在别处不合法（两者
  在哪里都是一直合法的普通语法），而是因为跳过一个 scpp 自己加的
  **运行时**检查，完全没有
  [§5.1](ch05-static-checks.md)-[§5.4](ch05-static-checks.md)
  那种"损坏的记账可能泄漏到外面代码"的风险，所以能保持无条件（完整
  理由见 [§5.8](ch05-static-checks.md)）。

## 1.2 没有函数级标记：需要就把整个函数体包起来

没有任何办法把一整个函数标记成豁免
[§5.1](ch05-static-checks.md)-[§5.4](ch05-static-checks.md) 的检查——
这是刻意的设计：没有关键字，没有文件级 pragma，什么都没有。如果一个
函数的实现确实需要在整个函数体里大范围用到
[§5.5](ch05-static-checks.md) 的那些操作（而不是一两个孤立的点），
表达方式就是把**整个**函数体包进一个 `unsafe { }` 块：

```cpp
int legacy_style_function(int* p, size_t n) {
    unsafe {
        // 整个函数体都在这里；§5.1-§5.4 照样全程生效
    }
}
```

所有权/move/别名/生命周期检查依然对块内每一条语句生效，跟
[§1.1](#11-unsafe--放宽了什么什么保持无条件) 说的完全一样——把整个
函数体包起来并不会改变这一点。这跟 Rust 自己在 2024 edition 收紧的
`unsafe_op_in_unsafe_fn` 是一个道理：哪怕是 Rust 的 `unsafe fn`，
现在也要求对它执行的具体操作显式套一层内层 `unsafe { }`，而它的函数
体从一开始、在任何 edition 下都从来没有豁免过借用检查。

## 1.3 `unsafe { }` 块（设计已定稿，尚未实现）

- **语法**：`unsafe` 后跟一个普通的花括号语句列表，可以出现在函数体内任何
  能写语句的地方：`unsafe { stmt; stmt; ... }`。v0.1 只支持这一种块语句
  形式。Circle/Safe C++ 还允许把 `unsafe` 直接标在单个条件表达式、
  match 分支体、构造函数的 mem-initializer 前面；scpp 因为还没有
  match 表达式、也没有构造函数，这些都先放着——以后各自的语法落地时，
  可以单独加成新的语法产生式，不影响这里定的规则。
- **作用域**：跟 Circle 不一样（Circle 里 `unsafe { }` 故意**不**开新的
  词法作用域），scpp 的 `unsafe { }` 是一个**普通的块**：跟普通的 `{ }`
  复合语句完全一样——里面声明的局部变量在 `}` 处结束生命周期，跟普通
  C++ 一样。如果某个 `{ }` 悄悄跟语言里其他所有 `{ }` 作用域规则不一样，
  就违反了 scpp 自己"看起来像 C++、不搞意外"这条原则。如果 `unsafe { }`
  里算出来的值需要在外面继续用，按普通 C++ 的写法，先在外层作用域声明，
  再在里面赋值即可。
- **解锁了什么**：[§5.5](ch05-static-checks.md) 列的那些操作，在
  `unsafe { }` 里恰好被解锁，且**仅**解锁这些：裸指针解引用/指针算术、
  `reinterpret_cast`/不兼容类型间的 C 风格强转、未加标签的 `union`
  成员访问、裸 `new`/`delete`、可变全局/静态变量访问、以及调用一个
  `extern "C"` 函数。这些里**目前只有两条在当前实现里能真正碰到**：
  裸指针 `*p` 解引用（`T*` 现在就是一个真实类型），以及调用一个
  `extern "C"` 函数（声明和调用现在都已经存在）。另外四条现在完全没有
  对应语法（`union`、`reinterpret_cast`、`new`/`delete`、全局变量在
  v0.1 里都还没有词法/语法支持）——这些语法落地之前无从"解锁"；等它们
  各自落地时，接入同一套机制即可。
- **`unsafe { }` 里还额外放宽了这些，但理由不一样**：`span` 的边界检查
  （[§8](ch08-open-questions.md) Q1）和整数溢出检查
  （[§5.8](ch05-static-checks.md)）也会被跳过——但跟上面那些不一样，
  这两个都不是"在 unsafe 之外就不合法"的操作（在哪里都是一直合法的
  语法）。它们是 scpp 自己加的**运行时**检查，碰巧完全没有下一条讲的
  "泄漏到外面代码"那种风险，这就是它们能加进这份放宽清单、又不
  跟下一条矛盾的原因。
- **什么保持一样严格**：[§5.1](ch05-static-checks.md)-
  [§5.4](ch05-static-checks.md) 里的所有东西——所有权/move 状态、别名
  XOR 可变、生命周期/悬垂检查、零初始化——在 `unsafe { }` 块的语句里
  **无条件**继续跑，没有例外。`unsafe { }` 是一个针对具体操作的窄口子，
  **不是**"这段区域别检查了"的总开关，实现时绝不能靠"跳过对这段语句跑
  movecheck"，只能靠放宽上面列的那几条具体检查。不管代码在不在
  `unsafe { }` 里面，`std::unique_ptr` 都不能被 double-move、别名
  XOR 可变都不能被违反——否则紧跟在块后面（仍然处在完整检查下）
  的代码就可能看到一个从来没被证明过安全的、已经损坏的状态。（裸指针
  本身不管在不在 `unsafe { }` 里都**依然不做** move/borrow 跟踪——这个
  不变，跟"裸指针本来就不是被跟踪的类型"这条现有规则是一回事。）
- **嵌套**：`unsafe { }` 可以嵌套在另一个 `unsafe { }` 里——是无害的，
  不是错误。
- **实现形状**（给做这个的人）：复用现有的 `StmtKind::Block` AST 节点
  （加一个 `is_unsafe` 标记），并且模仿现有的 `ScopeExit` MIR 模式，加一对
  对称的标记语句（比如 `UnsafeEnter`/`UnsafeExit`）把这个块 lower 出来的
  语句括起来。movecheck 的遍历只需要维护一个简单的嵌套计数器
  （`UnsafeEnter` 时 +1，`UnsafeExit` 时 -1），跟现有遍历逻辑并行即可——
  这个计数器**不需要**像 `DataflowState` 那样跨 CFG 分支做 join，因为它
  在每个程序点上是纯词法/结构性的事实，不是流敏感的（`if` 的每个分支
  在走到汇合点之前，自己内部的 `unsafe { }` 块早就已经闭合了）。上面
  提到的两条目前能碰到的检查，都在 `counter == 0` 时才生效；`span` 的
  边界检查和整数溢出检查（等实现的时候）也用同一个计数器，不需要另外
  搞一套机制。
