# 开始上手

scpp 是一门编译型语言。本章的目标很简单：先把编译器构建出来，写一个极小的程
序，把它编译并运行起来。我们暂时不追求一次讲完所有规则；先让一个真实程序跑
起来更重要。

## 你需要什么

如果你是从源码仓库开始跟做，需要准备：

- 一份 `scpp` 仓库副本；
- CMake 和 Ninja；
- 一套较新的 Clang/LLVM 工具链。

## 先把编译器构建出来

在仓库根目录运行：

```sh
cmake -S . -B build -G Ninja
cmake --build build
```

构建完成后，编译器可执行文件就是 `./build/scpp`。如果你之后把 `scpp` 安装到
了 `PATH` 上，那么本书中的同类命令只需要把前面的 `./build/` 去掉即可。

## 你的第一个程序

创建一个名为 `hello.scpp` 的文件：

```cpp
extern "C" int puts(const char* s);

int main() {
    [[scpp::unsafe]] {
        puts("Hello, world!");
    }
    return 0;
}
```

然后编译并运行它：

```sh
./build/scpp hello.scpp
./a.out
```

你应该会看到：

```text
Hello, world!
```

## 你刚刚已经见到了什么

这个小程序已经暴露出几件很重要的事情：

- `int main()` 是程序入口。
- 字符串字面量沿用你熟悉的 C++ 双引号写法。
- `extern "C" int puts(const char* s);` 声明了一个来自 C 库的函数。
- 这里需要 `[[scpp::unsafe]]`，因为调用一个无法被 scpp 静态验证的外部函
  数，在 scpp 里属于必须显式标出的边界。后面我们会专门讲这件事；现在你只需
  先把它理解成：这里有一小块“编译器需要你自己担保”的代码。
- `return 0;` 用来向操作系统报告“程序成功结束”。

第一章做到这里就够了：你已经亲手构建了编译器，也运行了一个真实的 scpp 程
序。

## 下一步

下一章继续保持“写出来、编译它、运行它”的节奏，但会把输出一行文字升级成一
个稍微更完整一点的小程序。

## 改写期间暂时保留的参考附录

下面保留的是较早写成、偏参考手册风格的内容。后面的章节目前仍然会链接到这些
材料，所以在教程化改写分批落地期间，它们暂时继续留在这里。

1. **视觉上就是 C++**：任何熟悉现代 C++ 的人第一眼就认为这是 C++。
2. **加法极简，而且要能干净地擦除回去**：只在绝对必要时引入新语法，而且
   前提是删掉它之后剩下的还是一个真正的 C++ 编译器能原样接受的普通文件。
   scpp **零新增关键字**：每一个 scpp 专有构造——`[[scpp::unsafe]]` 这个
   退出机制（见 [§1](ch01-safety-context.md)），跟 `[[scpp::lifetime(name)]]`
   生命周期分组（见 [§5.3](ch05-static-checks.md)）一样——都拼写成 `scpp`
   命名空间下的一个 attribute，因为真正的 C++ 编译器本来就会解析、并且
   悄悄接受一个它自己不认识的 attribute——这跟一个全新的关键字或者运算符
   token 不一样，编译器压根没法越过后者继续解析下去。把一份合法 scpp
   文件里每一个 `scpp` 命名空间下的 attribute 都去掉，剩下的就是原封不动
   的标准 C++——**scpp = C++ + 一个检查器**，不是一门有自己语法的新语言。
   这也是为什么 scpp 最终放弃了类似 Rust `?` 的错误传播运算符、改用普通
   `if`/`else`（见 [§5.6](ch05-static-checks.md)/[§8](ch08-open-questions.md)
   Q8）：运算符 token 跟 attribute 不一样，没有"悄悄忽略"这条退路。
3. **复用已知语法、重新赋予语义**：`std::move()`、`T&`、`unique_ptr`、`span`
   等既有写法，被赋予更强的静态含义（所有权/借用/生命周期），
   但不改变它们对用户的"外观"。
4. **安全是默认状态；`[[scpp::unsafe]]` 是唯一的退出方式，而且是局部的、
   可组合的**：每个函数都无条件受检查（见 [§5](ch05-static-checks.md)），
   不需要任何按函数/按文件的标注来开启。`[[scpp::unsafe]] { }` 是一个词法
   作用域块，可以嵌套、局部可逆，只放开 [§5.5](ch05-static-checks.md) 里
   那份固定、列举出来的操作清单——它从来不是一个能把整个函数或整个文件
   检查关掉的开关。这是对早先设计（一个 `safe` 关键字标注哪些函数 opt-in
   检查，没标的函数完全不检查）的刻意反转——理由见
   [§8](ch08-open-questions.md) Q13。
5. **健全性优先于兼容性**：宁可暂时"尚未支持"某语法，也不
   放行一个不健全的检查。100% C++ 兼容不是目标。
6. **可擦除性压倒人体工程学和功能完整性**：目前没有别的给 C 系语言补
   Rust 级别健全性的尝试，能在保留第 2 条这条可擦除性质的**同时**还真
   做到——保持可擦除的（比如微软的 SAL 标注）力度都远不如 Rust；真做到
   健全的（Sean Baxter 的 Circle/Safe C++）需要自己的编译器，彻底放弃了
   可擦除性；处在中间地带的（Checked C）恰恰在需要新语法表达的那些构造
   （指针类型、边界标注）上放弃了可擦除。**两头都要**，是 scpp 真正的
   赌注和差异化所在。所以以后任何一个新特性，如果非要引入全新的、不可
   擦除的语法才能完整表达、或者才能做到 Rust 那种人体工程学，答案**不是**
   加这个语法：想办法用已经能干净擦除的写法（哪怕啰嗦一点）表达同样的
   东西，或者干脆先不支持这个特性，等想到办法再说（`??` 运算符就是这条
   原则第一次具体落地的例子，见 [§8](ch08-open-questions.md) Q8）。啰嗦、
   功能暂缺，都是能接受、能挽回的代价；可擦除性被打破，不能接受。
7. **正式名字：SCPP26。参照的 C++ 标准：C++26。** scpp 的正式版本化名字，
   直接把它跟参照的标准绑在一起。总目标：**默认支持 C++26 的全部语法**，
   只排除一小撮明确列举出来的、真实 C++ 自己已经承认有缺陷的情况（有据
   可查的坑、明确被 deprecate 的写法，或者类似情况）——绝不是因为 scpp
   自己的设计更偏好某种更显式/不一样的写法（参见
   [§5.12](ch05-static-checks.md) 的 `this` 必须显式 capture 这条规则，
   建立在真实 C++20 自己的 P0806R2 deprecation 之上，是本文档里其它所有
   限制都应该照着走的范本）。另外，本文档里凡是说某个名字/特性是
   "真实的"/"已经标准化"的 C++（比如 `<cstdint>`、`<stdfloat>`、
   `std::expected`、C++20 modules）、还是"还不是标准"（比如
   `int128_t`/WG21 P1467——见 [§6](ch06-safe-subset.md)），分界线**就是
   C++26**：2025 年 6 月 Sofia 会议功能冻结，2026 年 3 月 London 会议
   完成最终投票。第 2 条的可擦除性和第 6 条"只用已批准的名字"这条规则，
   检验基准都是这一版。以后新标准批准了，这条基准要跟着往前挪。
8. **任何编译器自己能控制的地方，绝不允许 UB——哪怕在 `[[scpp::unsafe]]`
   里也不行。** 开发者必须永远能知道自己代码的确切行为。真实 C++ 里，哪怕
   程序什么"解引用一个乱指针"这种过分的事都没干，某些操作也是未定义的——
   有符号整数溢出就是最典型的例子（见 [§5.8](ch05-static-checks.md)）：
   编译器有权假设它永远不会发生，然后把（不知情地）依赖了这个假设的代码
   编译错——纯粹是因为标准这么写了，不是因为这件事没法定义清楚。scpp
   绝不给自己这张牌：只要一个操作的"是否有定义"是 scpp 自己 codegen 能
   选择的（整数溢出是第一个具体案例），要么无条件继续检查（默认状态
   下），要么在 `[[scpp::unsafe]]` 里把它钉死成一个具体的、确定的、写
   清楚的结果（比如保证的补码 wrap）——绝不是真正的 UB。这条不适用于
   （也没法消除）另一种 UB——来自**外部、没法验证的前置条件**那一种
   （解引用一个裸指针，只有指针真的有效才是健全的，这个没有任何静态
   分析能在一般情况下证明）——这一类上，`[[scpp::unsafe]]` 的含义跟
   Rust 的 `unsafe` 完全一样。
