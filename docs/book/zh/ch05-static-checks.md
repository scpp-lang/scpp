# 静态检查（健全性核心）

第 4 章讲的是 scpp 把值分成哪些类别；这一章讲的是另一半：**编译器在普通代码里，
到底会为这些值证明什么。**

最短但准确的总结是：

- safe scpp 代码默认就受检查；
- `[[scpp::unsafe]]` **不会**把这些检查整体关掉；
- 它只会重新放开一小份“检查器本身无法自行证明”的操作清单。

如果你熟悉 Rust，把 Rust 的所有权模型当直觉来理解就对了。scpp 想做的是：
在复用 C++ 外表语法的同时，拿到那套默认安全的静态保证。

## 5.1 所有权与移动（Move / Ownership）

在 scpp 里，每个非平凡值同一时刻都只有一个拥有者。对 `class` 值以及
`std::unique_ptr<T>` 这样的库类型来说，所有权通过 `std::move(x)` 显式转移。

一次 move 之后：

- 目标成为新拥有者；
- 源绑定进入 **moved-out** 状态；
- 在重新赋予新值之前，再读这个源绑定就是编译错误。

这也是析构为什么可以变得可预测：离开作用域时，仍然处于 initialized 状态的值
会被销毁；已经 moved-out 的值则不会。

```cpp
import std;
extern "C" int puts(const char* s);

int main() {
    std::unique_ptr<int> owner = std::make_unique<int>(9);
    std::unique_ptr<int> next = std::move(owner);

    if (*next == 9) {
        [[scpp::unsafe]] {
            puts("ownership moved exactly once");
        }
    }
    return 0;
}
```

输出：

```text
ownership moved exactly once
```

如果你来自 C++，这里有两个点特别值得记住。

第一，`std::move` 不再只是“也许会触发某个重载的一次 cast”。在 scpp 里，它是
所有权模型的一部分：写下它，就等于明确告诉编译器“这个绑定从这里开始不再拥有值”。

第二，[§4.1](ch04-struct-vs-class.md) 里的 `struct` 不走这套规则。因为 `struct`
按定义就是平凡类型，它们是按值复制，而不是进入 move 追踪。

## 5.2 借用与别名（Borrow / Aliasing）

scpp 的借用规则和 Rust 的核心规则是一回事：**别名 XOR 可变性**。

任意时刻，一个 place 只能处于下面两种状态之一：

- 拥有任意多个共享借用（`const T&`）；或者
- 拥有恰好一个可变借用（`T&`）；

两者不能同时成立。

这个规则是记在“根 place”上的，而不是记在“恰好通过哪段语法得到的引用”上。
v0.1 因此故意采用保守策略：借用 `obj.left` 和借用 `obj.right`，仍然会被视为
借用了同一个根对象 `obj`。这会损失一些精度，但模型更简单，也更容易保持健全。

## 5.3 生命周期（Lifetime）

借用不能活得比被借用值更久。这就是生命周期规则的一句话版本。

而对日常可用性最重要的一点是：v0.1 已经做到了**基于最后一次使用**的生命周期
推断。一个借用会在“最后一次使用之后”结束，而不是必须一直等到词法作用域的
右花括号。

```cpp
extern "C" int puts(const char* s);

int main() {
    int value = 5;
    const int& left = value;
    const int& right = value;
    int total = left + right;

    value = 9;

    [[scpp::unsafe]] {
        if (total == 10 && value == 9) {
            puts("borrows ended after their last use");
        }
    }
    return 0;
}
```

输出：

```text
borrows ended after their last use
```

执行 `value = 9` 时，`left` 和 `right` 还**在作用域里**，但它们已经**不再活跃**。
这正是 scpp 能接受很多自然代码写法的关键：它不会傻等到整个 block 结束。

### 跨函数生命周期

对“返回一个引用”这种情况，本书的完整设计会用 `[[scpp::lifetime(name)]]`
在必要时给参数分组命名。但 **当前编译器** 还只实现了比较小的一部分：

- 要么只有一个合格的引用形参；
- 要么是方法里的隐式 `this` 这一种常见情况。

参数上的 `[[scpp::lifetime(name)]]` 目前会被解析，以保证前向兼容；但真正完整的
“多组参数 / 指定返回组”机制暂时还没有强制执行。

## 5.4 初始化（Initialization）

scpp 没有“也许未初始化的局部变量”这种状态。一个没有显式初始化器的局部变量或
成员，会从一个定义良好的零值开始。

- 整数变成 `0`；
- `bool` 变成 `false`；
- 浮点变成 `0.0`；
- 裸指针变成 `nullptr`；
- 聚合类型则按字段递归清零。

```cpp
extern "C" int puts(const char* s);

struct Totals {
    int ok;
    int bad;
};

int main() {
    Totals totals;

    [[scpp::unsafe]] {
        if (totals.ok == 0 && totals.bad == 0) {
            puts("locals start from zero");
        }
    }
    return 0;
}
```

输出：

```text
locals start from zero
```

这个选择和 Rust/C++ 都不一样：Rust 会静态拒绝未初始化读取，C++ 则允许这里
掉进 UB。scpp 的答案是：默认值永远定义明确、结果可预测。

## 5.5 禁止项（除非在 `[[scpp::unsafe]]` 里）

`[[scpp::unsafe]]` 是**窄**的。它只重新放开一份固定列表，不会让所有权/生命周期
检查整体停机。

主要受管制的操作有：

- 裸指针解引用和指针算术；
- `reinterpret_cast` 与不兼容的 C 风格 cast；
- `union` 成员访问；
- 裸 `new` / `delete`；
- 可变全局 / 静态变量访问；
- 调用一个没有函数体的 `extern "C"` 声明；
- 调用一个声明成 `[[scpp::unsafe]]` 的函数；
- 通过 `[[scpp::unsafe]]` 函数指针发起间接调用。

```cpp
extern "C" int puts(const char* s);

int main() {
    int value = 42;
    int* p = &value;

    [[scpp::unsafe]] {
        if (*p == 42) {
            puts("unsafe is narrow and explicit");
        }
    }
    return 0;
}
```

输出：

```text
unsafe is narrow and explicit
```

这个例子很能代表 scpp 的整体设计。`&value` 这一步在安全代码里完全合法；
但当你准备**相信**这个指针、真正去解引用它时，就必须在本地明确写出来。

同样重要的是 `[[scpp::unsafe]]` **不会**做什么：

- 它不会允许 use-after-move；
- 它不会允许冲突借用；
- 它不会允许悬垂引用；
- 它不会把未初始化存储重新带回来。

这些检查在任何地方都继续生效。

## 5.6 可恢复错误：`std::expected<T, E>`

scpp 没有异常。可恢复错误的设计答案是 `std::expected<T, E>`。

这意味着：

- bug / 契约违例一类情况直接 abort；
- 可恢复失败通过普通返回值向上传播；
- 忽略一个 `expected` 结果会是硬错误，不只是 warning。

**当前编译器状态：** 这套设计已经定下来了，但 `std::expected` 还没有在当前
编译器 / stdlib 里实现。今天如果直接写它，仍然会被拒绝。

## 5.7 取地址（`&expr`）与裸指针

scpp 在边界问题上最重要的决定之一是：**形成**一个裸指针是安全的，**相信**
一个裸指针则不是。

- `&expr` 永远合法，并且会得到 `T*` 或 `const T*`；
- `T* -> const T*` 这条单向隐式转换仍然存在；
- 通过 `const T*` 写是普通类型错误，哪怕在 `[[scpp::unsafe]]` 里也一样；
- 裸指针解引用受 [§5.5](#55-禁止项除非在-scppunsafe-里) 约束。

这条分工让安全代码仍然可以准备好 C API 需要的 out-parameter，而不用假装裸指针
本身也参与借用追踪。

## 5.8 整数溢出

scpp 默认会检查整数 `+`、`-`、`*` 的溢出。

- 在普通代码里，溢出会 abort；
- 在 `[[scpp::unsafe]]` 里，这些运算会变成“不检查、但定义为 wrap”，而不是
  退化成 C++ 风格 UB；
- 除以 0、模以 0、以及 `INT_MIN / -1` 在任何上下文里都会 abort。

从概念上看，这和 `span` 的边界检查属于同一类：它们都是语言插入的运行时检查。
之所以允许在 `[[scpp::unsafe]]` 里跳过，是因为跳过它们不会破坏编译器的静态
所有权账本。

## 5.9 方法与 `this`

方法可以理解成：编译器给它隐式加了一个引用形参 `this`。

- 非 `const` 方法会独占借用接收者；
- `const` 方法会共享借用接收者；
- 它们再和普通引用遵守同一套别名规则。

所以，“先借用某个 public 字段，再调用同一个对象上的可变方法”在 v0.1 里会被
拒绝：这两个动作最后都落在同一个根对象上。

## 5.10 函数重载

scpp 保留普通 C++ 的重载语法，但它先拿掉了那个让 C++ 重载集变得很“玄学”的
东西：标量类型之间的隐式转换。

核心规则很简单：**重载只靠参数列表区分，永远不靠返回类型；一个候选只有在
参数类型精确匹配时才算可行。**

这条规则基本解释了这一节剩下的大部分内容。

- **只看精确类型匹配。** 因为 [§6](ch06-safe-subset.md) 已经规定不同 scpp 标量
  类型之间没有隐式转换，所以 scpp 不需要 C++ 那套 Exact Match > Promotion >
  Conversion > ... 的等级排序。一个调用要合法，每个实参类型都必须和对应形参
  类型完全一致；如果没有匹配项，就需要在调用点显式 cast。
- **这比真实 C++ 刻意简单。** 在 C++ 里，提升的目标是特定的
  `int`/`unsigned int`/`double`，不是“离它最近的更宽类型”；而两个都只是
  conversion 级别的候选，依然可能直接歧义。scpp 干脆不做这种猜测。
- **按值 / 按引用是另一条独立维度。** `f(T)`、`f(T&)`、`f(const T&)` 是三个
  合法且不同的重载。在 scpp 里，这三个写法对应三种完全不同的所有权语义：
  取走所有权、可变借用、共享借用。一个裸的命名左值只能匹配引用版本；像
  `std::move(x)` 或临时值这样的“新鲜值”才会让按值版本可行。如果一个可变左值
  同时匹配 `T&` 和 `const T&`，那么和普通 C++ 一样，`T&` 胜出。
- **作用域规则仍然跟 C++ 一样。** 内层声明会隐藏整个外层重载集；`using foo::bar;`
  会把当时可见的全部 `bar` 重载一起导入，而不是只导入某一个。
- **v0.1 没有 ADL 这层复杂度。** 一个非限定调用可见哪些候选，只由普通词法查找
  和 `using` 声明决定。
- **给重载函数取地址时，由目标类型决定选哪一个。** 目标函数指针类型会选出那个
  形参类型列表和返回类型都精确匹配的重载。到了 [§5.16](#516-函数指针function-pointers)
  这里还会再重要一次，因为函数指针类型自己也可能带 `[[scpp::unsafe]]` 资格。

所以，实践里的读法可以非常朴素：如果你只看源码里已经写出来的实参类型，就能
判断最终会选哪个重载，那你就是在按 scpp 期望的方式使用重载。


## 5.11 泛型函数与 Concept

泛型函数是 scpp v0.1 对“编译期多态”的回答。表面语法直接复用 C++20 的
`concept`/`requires`，但语义刻意偏向“受检查的泛型代码”，而不是“先写模板，
等实例化时再看会不会炸”。

这里最重要的是三条规则。

- **Concept 满足关系是 structural，不是 nominal。** 一个类型之所以满足某个
  concept，是因为它拥有所需的操作；scpp 没有 Rust 那种 `impl Trait for Type`
  式的显式名义性声明。这不是漏掉了，而是设计取舍：Rust 的 nominal trait 模型
  确实能避免“意外满足”这一类问题，但真实 C++ 没有现成的 `impl`-like 语法可借，
  scpp 也不想为此发明一套不可擦除的新语法，所以在这个问题上沿用了 C++ 的
  structural 模型，除非未来某个 C++ 标准自己先提供可借用的 nominal 机制。
- **受约束的泛型函数体只在它自己的定义处检查一次。** 在函数体内部，参数类型被
  当成抽象的，只允许 concept 的 `requires` 表达式真正保证过的那些操作。这和
  普通 C++ 模板是一个实质性的语义差别：scpp 不希望“函数体是否成立”取决于未来
  哪个具体实例化刚好触发了它，所以拒绝那种“这份模板正文只有某些实例化才碰巧
  合法”的检查风格，更接近 Rust 的 trait bound 心智模型。
- **实例化仍然是单态化的。** 这里没有 vtable，也没有运行时分派成本；泛型函数
  仍然是零成本抽象。

还有几个实践中很关键的细节。

- **表面语法上，concept 是可选的。** 从设计上说，裸的泛型参数表示“这个类型是
  opaque 的”：你可以移动它、存储它、透传它、返回它，但不能对它调用方法，也不能
  假设任何运算符存在；只有写出 concept 名字，才会解锁该 concept 真正保证的能力。
- **当前编译器状态。** 长期设计里，缩写形式的裸 `auto` 参数，以及带 concept 约束的
  参数包，都是这节语义的一部分；但当前编译器还没有实现这两个写法。今天真正可靠的
  工作写法仍是完整 header 形式（`template<typename T>`，可再配 `requires`），
  所以多数可运行示例都会写成那个样子。
- **参数包分成两种能力层级。** 按设计，缩写形式的 pack（例如
  `Concept auto&... args`）只该通过 fold expression 使用，因为 fold 对任何 pack
  长度（包括 0）都有唯一含义。完整 header 形式（`template<typename... Args>`）
  则还可以把 `args...` 再转发进别的调用或构造点。scpp 不想支持的，仍然是函数体
  里那种“拆出一个参数、剩下递归”的经典可变参模板写法，因为那会把语义重新推回到
  实例化时检查。
- **完整 header 形式拥有额外表达力。** 多个类型参数、显式但不参与推导的模板实参、
  以及“只出现在返回类型里的泛型函数”，都属于这一种写法；后面几节一旦需要超过
  “这里有一个受约束参数”的能力，也都是依靠它。

所以，对熟悉 C++ 的读者来说，最重要的心智切换是：*外表像 C++20 concept，
但函数体的检查契约更接近 Rust 的 trait bound，而不是传统模板实例化 folklore。*


## 5.12 闭包（Lambda 表达式）

lambda 表达式直接复用 C++ 语法，但 scpp 用最朴素的方式解释它：一个 lambda 会
下沉成编译器合成的匿名 `class`，每个 capture 对应一个字段，函数体则进到
`operator()` 里。

这点很重要，因为它意味着 lambda **不会**得到第二套特殊的所有权模型。

- **按值 capture 会变成普通拥有型字段。** 移动闭包就是移动这些字段；销毁闭包就
  是销毁这些字段。
- **按引用 capture 会变成引用类型字段。** 这会让闭包值本身也进入生命周期追踪，
  和任何其它“内部存了引用”的对象一样。
- **调用运算符按普通方法来检查。** receiver 借用、形参借用、返回值生命周期、
  move 检查，全都复用本章其它地方已经讲过的同一套机制。
- **`this` / `*this` 必须显式 capture。** 裸的 `[=]` 或 `[&]` 在 scpp 里不会
  隐式抓取 `this`，哪怕老 C++ 代码里这种写法很常见。scpp 想让这条生命周期边界
  明明白白地写在表面上。

所以，对 lambda 最安全的读法就是：如果你能把它解释成“一个小小的匿名 `class`”，
那你的理解方式就和检查器是一致的。


## 5.13 生命周期泛型参数（Lifetime-Generic Parameters）

这是 [§5.3](#53-生命周期lifetime) 里“具名生命周期分组”机制第一次需要一个小扩展
的地方。

它要解决的典型场景，长得很像 Rust 的 `thread::scope`：某个库函数在**自己的函数体
内部**临时造出一个短生命周期的值，然后想把指向它的引用传给回调。这个生命周期是
被调用方临时决定的，回调作者事先根本不可能把它写出来。

scpp 对这个模式的处理方式，是在现有 `[[scpp::lifetime(name)]]` 机制上做一个定点
扩展，而不是整套引入 Rust 那种 lifetime parameter 语法类别。

- **`[[scpp::lifetime(generic)]]` 是保留的分组名。** 它的意思是“一个全新的、由
  编译器合成的分组，用户代码不能再用别的地方去命名它，也不能把它和任何别的分组
  unify 起来”。这会立刻带来一个约束：在 callable 自己的函数体里，带 `generic`
  标签的引用可以被同步地读取、调用方法、再转发给兼容参数；但它不能被塞进别的具名
  分组、不能被返回、也不能被藏进更长寿命的状态里。
- **`requires` 表达式可以精确探测这个性质。** 如果一个 concept 的 probe parameter
  带上 `[[scpp::lifetime(generic)]]`，那么“满足这个 concept”的意思就是：目标
  callable 自己对应的那个形参也真的是 lifetime-generic，而不只是“恰好能接这个
  测试调用”而已。这是在一个本来就存在的 C++ 语法位置上，附着了一条新的
  concept-checking 语义。
- **对这种 callable 的调用会得到一条定点豁免。** 一旦类型系统已经知道某个 callable
  的参数是 lifetime-generic，调用方就可以把一个“在调用方自己函数体里新鲜发明
  出来的引用”传进去。之所以 sound，正是因为前面两条已经保证了 callee 不能把它
  藏到任何会活过这次调用的地方。

这覆盖的是 Rust 靠 higher-ranked trait bound 表达的那类 soundness pattern，但
做法仍然是延伸 scpp 已有的“具名分组”设计，而不是反过来再引入一整套 `'a` 风格
的生命周期泛型语法。

**当前编译器状态。** 上一版新增的这个 caveat 值得保留：今天编译器真正强制执行的，
仍然只是 [§5.3](#53-生命周期lifetime) 里那个较旧的“单个引用形参 / 隐式 `this`”
子集。这里的 `[[scpp::lifetime(generic)]]` 仍然是目标设计，不是已经完整落地的实现。


## 5.14 泛型类型

泛型 `struct` / `class` 把 [§5.11](#511-泛型函数与-concept) 的编译期多态故事，从
函数扩展到了类型定义本身。

第一条该记住的仍然是第 4 章那条分界线：`struct` 和 `class` 的差别，在泛型里一样有效。

- **泛型 `class` 可以把类型参数裸着写。** 它的意思和泛型函数里完全一样：这个类型只
  保证能被移动、存成字段、透传、返回；若没有某个成员自己加 `requires` 子句，就不能
  假设它支持任何方法或运算符。
- **这些 `requires` 约束是按成员逐个加的，不是整类一刀切。** 同一个泛型类型里，某个
  构造函数或方法可以要求 `std::totally_ordered<T>`，另一个成员则可以只依赖裸参数的
  最小能力。
- **泛型 `struct` 不能把类型参数裸着写。** 因为 `struct` 的 triviality 是整个类型的
  布局性质，不是某个成员方法局部能补救出来的，所以泛型 `struct` 必须把参数约束到
  足以保证每个字段仍然 trivial。

而这节最重要、最可操作的一条规则，是上一版丢掉的那一条：**如果你要写一个
Tuple-like 的变参泛型类型，scpp 只支持“递归继承”这一个模式；不支持把参数包直接
展开成成员列表。** 原因很直接：到目前为止，没有任何已采纳的 C++ 标准给出真实可用的
`Ts... members;` 语法，而 scpp 的擦除原则要求这些源码仍能被普通 C++ 编译器正常解析。

所以，受支持的形状就是下面这两个固定步骤：

```cpp
template<typename... Ts>
class Tuple;

template<>
class Tuple<> {};

template<typename Head, typename... Tail>
class Tuple<Head, Tail...> : private Tuple<Tail...> {
    Head head;
};
```

应该这样读：

1. `Tuple<>` 是空参数包的 base case；
2. `Tuple<Head, Tail...>` 自己存一个字段，其余元素交给基类那一层继续存。

这就是读者今天在 scpp 里真正该照着写的 variadic owning type 模式。

还有几条边界也值得说清楚：

- **按索引访问，目标是复用 base-class deduction。** 常见的 `get<I>` 写法，
  本质上是让目标基类特化被推导出来，而不是让 accessor 自己在运行时递归。
- **非类型模板参数只支持标量类型。** 这样就可以绕开 C++ 那个 class-typed
  structural value 世界里更复杂的一大坨规则。
- **任意特化仍然不在范围内。** 当前支持的变参模式，就是上面这个固定的
  空包 / Head+Tail 递归，不是通用 specialization 机制。

所以，这一节最后真正该留下来的实现级结论是：如果你想写一个可工作的泛型
`Tuple`，就按“空包基类 + 递归继承”去写。


## 5.15 线程安全的结构性属性：`[[scpp::thread_movable]]`、`[[scpp::thread_shareable]]` 与 `[[scpp::thread_movable_if(a, b)]]`

这些 attribute 是 scpp 用来表达“这个值能不能安全地跨线程移动 / 共享”的
结构性词汇。

在概念上，它们扮演的角色和 Rust 的 `Send` / `Sync` 很像：

- 库 API 可以要求某个实参必须 thread-movable 或 thread-shareable；
- 类型也可以按结构规则自动推导出这些属性；
- 当默认结构规则过弱或过强时，attribute 提供一个显式覆盖点。

内置谓词 `scpp::is_thread_movable(T)` 和 `scpp::is_thread_shareable(T)`
则让泛型代码可以查询这些性质。

## 5.16 函数指针（Function Pointers）

scpp 复用普通 C/C++ 的函数指针语法，但在类型本身上多记了一件至关重要的事情：
一个“指向函数的指针”类型，要么带 `[[scpp::unsafe]]` 资格，要么不带。

之所以要有这一个额外 bit，是因为语言已经规定了：某些函数调用本身需要本地担保。
如果你把一个 `[[scpp::unsafe]]` 函数、或者一个没有函数体的 `extern "C"` 声明
取地址之后，得到的却是一个普通、看起来完全无害的函数指针值，那么程序就可以先把它
存起来，再通过这个变量去调用，悄悄绕过原本的 unsafe gate。

所以，规则必须是：**函数指针类型自己也要记住“通过它调用是否受管制”这件事。**

- **拼写仍然是实打实的 C++ 语法。** 标记写在指针声明符上，也就是 `*` 的后面：
  ```cpp
  int (* [[scpp::unsafe]] up)(int, int);
  int (*                  sp)(int, int);
  ```
  这本来就是 C++ declarator grammar 里已有的 attribute 位置，所以 scpp 不需要
  为此再造新语法。
- **取地址时，哪种 flavor 会自动决定。** 普通函数的地址得到普通函数指针；
  `[[scpp::unsafe]]` 函数，或者没有函数体的 `extern "C"` 声明，取出来的地址则是
  unsafe-qualified 函数指针。
- **转换只有单方向：普通 → unsafe-qualified。** 把普通函数指针塞进一个
  unsafe-qualified 槽位是无害的，因为这只会给调用方增加义务；反方向则会抹掉一个真实
  的安全要求，所以必须拒绝。
- **通过指针调用时，看的是指针自己的类型。** 普通函数指针可以自由调用；
  unsafe-qualified 函数指针则正式加入 [§5.5](#55-禁止项除非在-scppunsafe-里)
  的受管制清单，必须在 unsafe context 里调用。
- **“调用函数指针”不是“解引用裸数据指针”。** `fp(args)` 和 `(*fp)(args)` 都是普通的
  函数指针调用语法；它们不会触发 [§5.7](#57-取地址expr与裸指针) 那条“解引用原始内存”
  规则。
- **函数指针仍然是 trivial 的。** 和任何其它裸指针一样，它们不带编译器跟踪的生命
  周期，可以自由复制，也可以合法放进 `struct`。
- **对重载函数取地址时，由目标类型选中那一个重载。** 目标函数指针类型会挑出那个
  形参类型列表和返回类型都精确匹配的版本。

成员函数指针（pointer-to-member-function），以及“返回函数指针的函数”，仍然不在 v0.1 范围内。
但对读者来说，真正可操作的工作规则很简单：如果按名字调用这个函数本来就需要本地
担保，那么把它塞进函数指针之后，这个义务也不会消失。


## 5.17 解引用运算符作用于 `class`

对用户自定义 `class` 来说，`*x` 和 `x->y` 不是神奇的逃生舱。scpp 故意让它们
保持“无聊”。

- **`*x` 只是方法调用语法糖。** 如果这个类声明了 `operator*()`，那么 `*x` 的检查
  方式就和 `x.operator*()` 完全一样，用的是普通方法那套 receiver 借用规则。
- **返回出来的引用服从普通生命周期规则。** 如果 `operator*()` 返回 `T&` 或
  `const T&`，那么对 `*x` 的借用会记在 `x` 的根对象上；只要这个借用还活着，
  move 或重新赋值 `x` 都会被拒绝，理由和任何其它“方法返回引用”场景完全相同。
- **`x->y` 仍然只是 `(*x).y` 的语法糖。** 一旦 `*x` 能工作，经由 `->` 的成员访问
  就会自动跟上；不需要再单独设计第二套所有权规则。
- **没有单独可重载的 `operator->`。** scpp 不再额外引入真实 C++ 那套第二协议，
  因为只靠 `(*x).y` 这一个重写，就已经拿到了有用的表面写法。

所以，最安全的理解方式是：这个特性只是让“像指针一样工作的类”接入既有语法糖，
并没有为借用检查新开一条绕行通道。


## 5.18 类型擦除调用包装器：`std::function` 与 `std::move_only_function`

scpp 提供两种拥有型、类型擦除的 callable wrapper：

- `std::function<Sig>`：面向可复制的目标；
- `std::move_only_function<Sig>`：面向只能移动的目标。

这里最关键的设计点是：**它们是库层抽象，不是那种很深的编译器魔法。** 只要泛型
`class` 模板已经能表达多个模板参数、对函数类型模板实参做偏特化、泛型构造函数、
以及方法里的参数包，这两个包装器就只是普通库代码。它们真正需要的 builtin 帮助，
无非就是把 `R(Args...)` 这种函数类型模板实参看成“返回类型 + 参数包”——这和真实
C++ 里偏特化对 `R(Args...)` 做模式匹配，本质上是同一个角色。

有几条语义规则值得明确写出来。

- **没有 empty / null 哨兵状态。** 和真实 C++ 的 `std::function` 不一样，scpp 的
  callable wrapper 一旦存在，就一定内部装着一个有效目标。如果程序需要“也许这里没有
  回调”，那这个 optionality 应该出现在外层类型里，比如 `std::optional<...>`。
- **move 复用普通 class 规则。** 移动一个 wrapper，就是把内部 callable 移到目标
  wrapper 里，并让源 wrapper 进入普通的 moved-out 状态；源对象不是一种“仍可继续用，
  只是内部为空”的特殊盒子。
- **`std::move_only_function` 从一开始就支持 cv/ref-qualified 签名。**
  `void() &` 表示目标必须能作为左值被调用；`void() &&` 表示目标必须能作为右值
  被调用；`void() const` 则表示目标必须支持 const 调用。
- **如果有 extract/release 操作，它必须是消费式的。** scpp 不想让“把目标抽走之后，
  盒子变成 empty sentinel”这件事重新潜入语义里。
- **只有当“擦除”本身就是目的时，才该用它们。** 如果一个函数只是想调用传进来的
  闭包，那么 [§5.11](#511-泛型函数与-concept) / [§5.12](#512-闭包lambda-表达式)
  那条泛型路径仍然更好：没有 erased dispatch，没有隐藏盒子，编译器还能完整看见
  具体类型。

所以，`std::function` 和 `std::move_only_function` 真正适用的场景，是你确实需要
“用一个拥有型字段或一个返回类型，去隐藏多个不同 callable 具体类型，但统一暴露成
同一个签名”。


## 5.19 `union` 与 `[[scpp::packed]]`

`union` 的存在是为了 FFI 和存储重叠场景，不是为了日常安全数据建模。普通安全
代码可以携带一个 `union` 值，但只要读写某个成员，就必须进入
`[[scpp::unsafe]]`，因为编译器不会追踪“当前活跃成员是哪一个”。

`[[scpp::packed]]` 则是显式的“匹配外部 packed 布局”属性，可挂在 `struct` 和
`union` 上。只有当外部 ABI 真的要求这样时才该用它，而不是把它当成通用性能技巧。

## 这一章真正该记住什么

如果第 5 章信息量太大，先只记住下面三条，再把剩下的内容当参考手册回来看：

1. **所有权显式移动。**
2. **借用必须同时满足 alias XOR mutability 和 lifetime。**
3. **`[[scpp::unsafe]]` 很窄，不会关掉检查器。**
