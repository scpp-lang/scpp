# 4. struct 与 class 的语义区分（内存布局 / ABI 固定）

这是与 C++ 的一个关键差异：C++ 里 `struct` 和 `class` 几乎等价（仅默认可见性
不同）；scpp 里两者语义完全不同。

## 4.1 struct：纯平凡数据（trivial aggregate）

- `struct` 只能包含**平凡（trivial）类型**的成员，递归定义如下：
  - 标量类型：`bool`、整型、浮点、`char`。
  - 裸指针 `T*`（不带编译器跟踪的生命周期；解引用仍需 `[[scpp::unsafe]] {}`，见
    [§2](ch02-boundary-rules.md)）。
  - 函数指针（不管是 unsafe-qualified 还是不是 unsafe-qualified 的
    指向函数的指针类型，见
    [§5.16](ch05-static-checks.md#516-函数指针function-pointers)）——
    同样不带编译器跟踪的生命周期。
  - 其他同样满足本规则的 `struct` 类型（递归）。
  - 平凡类型的定长数组。
- 以下类型**禁止**作为 `struct` 成员，必须改用 `class`：
  - 引用 `T&` / `const T&`。
  - `std::span<T>`、`std::string_view`（带生命周期检查的借用视图）。
  - `std::unique_ptr<T>`、`std::shared_ptr<T>`、`std::vector<T>`、
    `std::string`，或任何参与所有权/借用检查的类型。
- 未通过此规则的 `struct` 定义是编译错误（提示"改用 class"），而不是静默
  降级。

**初始化**：`struct` 局部变量/成员若未显式初始化，编译器保证整个对象**按
位清零**（zero-init）。标量成员等价于 `0` / `false` / `0.0`；裸指针成员
等价于 `nullptr`。这不是 `struct` 的特例，而是 scpp**所有类型**共享的通用
规则——见 [§5.4](ch05-static-checks.md) 的完整说明。

**Copy 语义**：`struct` 值可以自由、隐式地按位复制，不参与
[§5](ch05-static-checks.md) 描述的 move/借用检查——因为它不携带任何
生命周期或独占所有权语义。这不是一个"可选的 Copy trait"（对比 Rust 需要
显式 `#[derive(Copy)]`）：一个类型只要声明为 `struct`，编译器就会验证并
保证它满足平凡性，`struct` 关键字本身就是显式声明。

## 4.2 class：拥有资源/参与检查的类型

- `class` 可以包含任意类型的成员，包括 `unique_ptr`、`vector`、`span`、
  其他 `class`，或本身携带生命周期/所有权语义的字段。
- `class` 参与 [§5](ch05-static-checks.md) 描述的所有权/移动/借用/
  生命周期检查；**不**保证 zero-init（需要显式构造）。
- **构造与析构**：一个 `class` 可以声明一个或多个构造函数（真实 C++ 的
  重载规则，[§5.10](ch05-static-checks.md)，直接适用——按参数列表区分，
  跟其它重载名字一模一样），也可以选择性地声明一个析构函数，语法照抄
  真实 C++。这不再局限于标准库提供的 `unique_ptr`（M2 里程碑最初的
  范围）——任何用户自定义的 `class` 现在都能拥有资源、自己定义清理
  逻辑。
- **Move 构造和 move 赋值永远不是用户写的——编译器总是自己提供**：
  一个程序如果自己声明 `ClassName(ClassName&&)` 或者
  `operator=(ClassName&&)`，会被拒绝——对**每一个** `class` 类型都
  一样，无条件。取而代之的是：每个 `class` 都会拿到一个编译器合成的
  move 构造函数和 move 赋值运算符，按声明顺序递归 move 每个成员
  （标量/裸指针/`struct` 成员按位搬，`class` 类型的成员走它自己的
  编译器提供的 move）——跟真实 C++ 自己的**隐式定义**move 构造/赋值
  一模一样，只是永远不能被用户覆盖。这是刻意的、永久性的限制，不是
  临时的，理由跟真实 C++ 的 move 构造函数是个公认的坑是一回事：一个
  手写的 move 构造函数，只要忘了重置源对象（比如取走一个裸指针字段后
  忘记把它清空），就会留下两个活着的对象拥有同一个资源，两个最终都会
  去释放它。先拿真实 Rust（`rustc`）验证过：Rust 压根没有"move 构造
  函数"这个概念——move 永远只是按位拷贝 + 编译期把旧绑定标记失效，
  没有任何 trait 或 hook 能定制它。需要在值换手时跑自定义逻辑，走的是
  `Clone`（逻辑随便写，但永远是显式 `.clone()` 调用，从不会被普通
  move/赋值隐式触发）——而且一个类型实现了 `Copy` 就不能再实现
  `Drop`（验证过：`rustc` 会硬拒绝，报 E0184），这一刀直接把"资源类型
  被隐式按位复制"这个真实 C++ 的隐式特殊成员函数规则至今都没堵上的坑
  （只声明析构函数，隐式声明的 copy 构造函数现在也还是只被标记
  deprecated，不是删除——[depr.impldec]）彻底堵死。move 永远只有一种，
  结构化、编译器自己的，不给作者自定义逻辑留出错的空间——这次决定的
  完整记录见 [§8](ch08-open-questions.md) Q14。
  - 被 move 走的对象会被置于 *moved-out* 状态
    （[§5.1](ch05-static-checks.md)），跟 move 走任何别的值一样——如果
    它有析构函数，之后也不会为它调用（[§5.1](ch05-static-checks.md)）
    ——这是编译期就决定好、直接不生成那次调用，不是手写 C++ 通常靠
    运行时判空来达到同样的安全效果。
  - 因为 move 永远是同一种编译器提供的、纯结构化的操作，自我 move
    赋值（`x = std::move(x)`）不需要像真实 C++ 那样额外判断
    `this != &other`：求值 `std::move(x)` 会先把 `x` 置于 moved-out
    状态，然后赋值操作自己"析构旧值"那一步才轮到执行——这时候按
    "moved-out 对象不析构"这条同样的规则，那一步天然就是空操作。
  - 一个带引用类型（`T&`/`const T&`）非 static 数据成员的 `class`，
    压根没有 move 赋值运算符——跟真实 C++ 自己的规则
    （[class.copy.assign]）一模一样，理由也一样（引用没法通过赋值
    重新绑定）——只有 move 构造函数（引用还是能在构造时绑定一次的）。
- **Copy 构造和 copy 赋值，跟 move 不一样，允许用户自己写**——真实
  C++ 语法照抄（`ClassName(const ClassName&)`/`operator=(const
  ClassName&)`），逻辑随便写。一个 `class` 只有在自己一个都没声明
  copy 构造函数、copy 赋值运算符、析构函数这三者的情况下，才会拿到
  编译器提供的 copy 构造函数；反过来，也只有一个都没声明 copy 赋值
  运算符、copy 构造函数、析构函数这三者，才会拿到编译器提供的 copy
  赋值运算符。也就是说，这三者里只要声明了任何**一个**，就足以让
  另外那个特殊成员函数不再自动生成——不存在"一个用户写、一个编译器
  自动给"这种混合状态；一个只声明了 copy 构造函数（没声明析构函数）
  的 `class`，如果还想要 copy 赋值运算符，也得自己声明，反过来也一样。
  这次决定的完整记录见 [§8](ch08-open-questions.md) Q15。
  - 这是故意比真实 C++ 自己的规则更严格：真实 C++ 只是把——当一个
    class 带用户声明的析构函数或者 copy 赋值运算符时——copy 构造
    函数标记为 *deprecated*（对称地，带用户声明的析构函数或者 copy
    构造函数时，copy 赋值运算符也标记为 deprecated），但照样隐式
    生成，而不是压根不生成（[depr.impldec]）。C++ 标准自己都说这只是
    权宜之计，不是最终答案："未来版本的 C++ 可能会规定这些隐式定义
    是 deleted 的"（[depr.impldec]/1）。scpp 干脆现在就把这个标准
    自己已经预告的未来提前实现：把真实 C++ 那种"用到时候才警告"变成
    这个函数压根不存在——用到时候是编译报错，不是警告。
  - 这就是 scpp 版本的 Rust `Clone`——复用真实 C++ 自己的 copy 构造/
    赋值写法，以及它原本的隐式触发方式（`Foo b = a;`、`b = a;`），
    而不是要求显式 `.clone()` 调用——这是刻意的，跟 move 不一样：move
    上 scpp 是原样照抄 Rust"完全不允许自定义逻辑"这条规则（见上面），
    copy 上没有。为什么 copy 能放行、move 不能：Rust 自己禁止
    `Copy`+`Drop` 同时存在，是为了防止一个拥有资源的类型被**悄悄地、
    隐式地**按位复制——而 scpp 不需要靠禁掉自定义逻辑就能达到同样的
    效果，办法是压根不给带析构函数的 `class` 自动生成 copy。一个拥有
    资源的 `class`，作者必须自己手写一份经过审视的 copy 构造/赋值
    （通常是给引用计数 +1，或者深拷贝一份自己拥有的数据，比如手写一个
    `shared_ptr` 那样的类型）才能重新拥有 copy 能力——不存在"资源被
    悄悄复制了一份，而作者从来没写过任何正确复制它的代码"这种场景。
  - 因为现在允许用户声明自定义逻辑了，自我 copy 赋值（`x = x`）的
    安全性**只**在编译器自动提供的那种情况下有保证（按上面的规则，
    这种情况永远不会跟用户声明的析构函数、copy 构造函数或者 copy
    赋值运算符共存，所以不管别不别名都没什么好双重释放的）——一个
    用户自己写的 copy 赋值运算符，如果需要，得自己判断
    `this != &other`，跟真实 C++ 一样，scpp 不限制它能做什么。
  - 跟 move 一样的引用字段例外：一个带引用类型非 static 数据成员的
    `class`，可以有编译器提供的 copy 构造函数，但永远没有编译器提供的
    copy 赋值运算符（用户自己写一个还是可以的，只要那个引用字段允许
    作者在函数体里实际做的事情）。
- **可失败的构造与析构**：
  构造函数/析构函数没有通道把 `std::expected<T, E>` 传回去——scpp 根本
  没有异常可以替代地抛出（见 [§5.6](ch05-static-checks.md)/
  [§8](ch08-open-questions.md)），这两个特殊成员函数也都不返回普通值。
  这不是一条需要额外强制的新规则，而是"scpp 哪里都没有异常"这个事实
  白送的推论。
  - 构造函数/析构函数仍然可以校验前置条件，但校验失败只能靠 abort 处理
    （归为 [§8](ch08-open-questions.md) Q3 意义上的"bug"）——不能产生
    可恢复错误。
  - 如果一个类型的构造确实可能因为可恢复的原因失败（文件不存在、输入
    非法……），就不应该通过构造函数暴露这种失败，而是改用一个普通的
    `static` 成员函数，返回 `std::expected<T, E>`，只有确定成功之后才
    调用那个"保证成功"的构造函数。推荐做法（只需要普通 C++ 的访问控制
    就能落实，不需要新机制）：把那个朴素构造函数设成 `private`，让外部
    只能走这个工厂函数——经典的 C++"named constructor idiom"
    （Marshall Cline 的 C++ FAQ），不需要任何新增的 scpp 语法。
- **访问控制**：真实的、不受限制的 C++ 访问控制——成员变量或者成员
  函数，各自可以是 `public`或者 `private`，任意搭配，跟真实 C++ 完全
  一样（`protected`还是跟继承一起搁置，见下面）。外部代码直接访问一个
  公开的成员变量（`a.b`），检查方式跟 `struct`的字段访问
  （[§5.2](ch05-static-checks.md)）完全一样：借用记在整个根对象上，
  保守处理——这是 `struct`本来就在用的简化处理，不是为这里新发明的。
  方法调用（`a.f()`）继续按 [§5.9](ch05-static-checks.md) 已有的
  "this 当引用参数"模型检查。这两者不需要任何新机制就能协同工作：
  两边最终都是把借用记在同一个根对象上，所以持有 `a.b`的借用，跟调用
  `a`上某个会修改状态的方法，会正确地互相冲突——跟任何两个冲突的、
  对同一个根的借用一样。
  - 类级别的常量因此可以就是一个普通的 `public` `static constexpr`
    成员，原样复用真实 C++ 语法——`constexpr`**变量**（不是
    `constexpr`**函数**）没有歧义，本来就完全支持
    （见 [§5.11](ch05-static-checks.md)）——或者，如果想把值本身
    藏起来不暴露在公开接口里，也可以用一个 `private`常量、配一个
    `static consteval` 函数暴露出来，运行时开销同样是零。
- **v0.1 没有继承**（暂时搁置，不是永久排除——见
  [§8](ch08-open-questions.md)）：`protected` 因此也不是一个被识别的
  访问修饰符，因为它只有相对于派生类才有意义。以后如果真要设计继承，
  这两个一起重新考虑。编译期多态（通过一套共享的、受检查的接口去调用
  形状不同的类型）不需要等继承——见
  [§5.11](ch05-static-checks.md) 里基于 `concept`/`requires` 的泛型
  函数。
- **内部可变性，第一阶段**（回答了
  [§8](ch08-open-questions.md) Q4 里 `Cell` 那一半，`RefCell` 那一半
  搁置）：scpp 复用真实 C++ 的 `mutable` 关键字，但语义比真实 C++
  **严格得多**（真实 C++ 里 `mutable` 字段完全不设防）。一个 `mutable`
  成员变量：
  - 必须是**平凡（trivial）类型**，套用 §4.1 已经给 `struct` 字段定好的
    同一条规则（标量、裸指针、其它平凡类型、这些类型的定长数组）——
    对应 Rust `Cell<T>` 要求 `T: Copy` 的那条限制；
  - 可以通过**任何** `this` 读写，不管是不是 `const`——`const` 不再
    拦这一个字段的直接访问；
  - **永远不能**被引用或者取地址（`T&`/`const T&` 绑定、`&expr`）——
    两种都是编译错误，无条件成立，不管在不在 `[[scpp::unsafe]] { }` 里都一样。
  正因为永远不可能有引用指向它，运行时压根不需要检查任何别名风险——
  读写就编译成普通的内存读写指令，跟 Rust 的 `Cell::get`/`Cell::set`
  一样便宜。这能覆盖常见场景（计数器、标志位、小的缓存值）；`RefCell`
  那种场景（借用非平凡内部状态的真正引用，运行时计数检查，违规就
  panic/abort）真实 C++ 里没有现成名字可以借用，需要真正的新机制
  （运行时借用计数器、RAII guard 类型）——留到以后单独一轮再定。
  `mutable` 在 `struct` 上没有意义（`struct` 压根没有 `this`/方法/
  const 访问控制这些概念），也不接受这个写法。

## 4.3 内存布局与 ABI（固定，不作为 impl-defined 留白）

scpp 把 `struct` 的内存布局**钉死**为目标平台的 **Clang ABI**：在相同
target triple 下，scpp 编译器产出的 `struct` 布局必须与 Clang 编译等价
C 结构体产生的布局逐字节一致。具体规则：

1. 成员按源码声明顺序排列，编译器**不做重排**。
2. 每个成员按其类型在目标 Clang ABI 下的对齐要求对齐；相邻成员之间按需
   插入 padding。
3. 结构体自身对齐 = 所有成员对齐要求的最大值；总大小向上取整为该对齐的
   倍数（保证数组场景每个元素都正确对齐）。
4. 第一个成员偏移量固定为 0（结构体地址即第一个成员地址）。

实现上：编译器生成非 packed 的 LLVM 具名 struct 类型，交给目标 target 的
`DataLayout` 计算布局——该 `DataLayout` 与 Clang 针对同一 target triple
使用的完全一致，因此自动获得与 Clang/C 兼容的布局，无需 scpp 自行定义
对齐算法。

**v0.1 明确不支持**：
- 位域（bit-field）——各平台/编译器版本实现差异大，暂不纳入。
- 压缩布局（等价于 `#pragma pack(1)` / `__attribute__((packed))`）——留待
  后续版本通过显式属性（如 `alignas` / `packed`）支持，映射到 LLVM 的
  packed struct 类型。
