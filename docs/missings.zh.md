# 在重写本书时注意到的类 Rust 能力缺口

此文件是内部积压清单，不是面向读者的书籍内容。

## 语言 / 标准库缺口

- 目前还没有类似 `std::string_view` 的借用文本视图，这使得讲解类似 Rust 的“借用文本切片”示例更困难。
- 目前还没有像 `.size()` 这样的普通成员调用语法；本书现在不得不把 `.size` 解释为 `std::span` 上的一个计算字段。
- `std::span` 目前只能由定长数组构造，不能像 Rust 那样从 `Vec<T>` 或数组来讲解切片时那样，由更丰富的自有容器构造。
- `std::span` 在构造后不能重新绑定，因此它的行为更像一个永久绑定的借用，而不是一个可以自由重新赋值的视图值。
- 没有独立的 `operator->` 模型；`x->y` 只是通过 `operator*` / `std::unique_ptr` 提供的语法糖，这相比 C++ 和一些 Rust 包装器模式，缩小了“智能指针”设计空间。
- 书里已经设计了跨函数的具名生命周期组，但当前编译器实现的仍只是这套叙事中的一个很小子集。
- 还没有语言级的枚举、模式匹配、`if let` 或 `let...else`，所以 TRPL 风格里关于带标签备选值的章节必须改写或推迟。
- 还没有像 `std::vector`、`std::string` 或哈希映射等价物这样的可增长标准容器；本书目前只能讲解定长数组和与 C 兼容的缓冲区。
- 还没有 `for` / range-for 循环；面向学习者的迭代讲解目前仍必须以 `while` 和显式索引为中心。
- `const T&` 形参绑定仍然拒绝 `double` 字面量，因此希望对标量调用也友好的泛型 API 目前不得不优先使用按值传参；一个明显的后果是，`std::print`/`std::println` 现在只有通过 `std::move(s)` 或新建一个临时值，才能接受裸 `std::string` 左值。
- v0.1 中没有继承或基于虚函数的动态分发，因此无法直接翻译 TRPL 中关于面向对象 / trait object 的讨论。
- 还没有 async/await、future 或 stream 相关叙事，所以 TRPL 的 async 章节目前在 scpp 中没有直接对应内容。
- 还没有宏系统，这使得整整一类 Rust 元编程示例无法进入任何近期的 scpp 书籍。

## 工具链缺口

- 目前还没有针对 `docs/book/` 的、类似 mdBook / Rustdoc 的自动代码块提取与编译测试 harness；示例目前只能通过单独的手工脚本或外部测试来验证。
- 还没有类似 `rustup` 的安装器或工具链管理器；目前“入门”流程仍然是“从源码构建，然后可选地用 `cmake --install` 把它安装到你的 `PATH` 上某个位置”。
- 还没有类似 Cargo / crates.io 的注册表、包发布或依赖获取工作流；当前项目构建只覆盖本地 manifest、路径依赖和 workspace，还不具备完整的包生态系统。
