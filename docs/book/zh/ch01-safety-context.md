# 第一个完整的小程序

想真正熟悉一门语言，最快的方式之一就是尽早写出一个“像完整程序”的东西，而
不是只看零散片段。本章我们来写一个很小的倒计时程序：它仍然足够短，你可以一
口气看懂；但它已经足够完整，能让你接触变量、循环和分支。

## 程序全文

创建 `countdown.scpp`：

```cpp
extern "C" int printf(const char* fmt, ...);
extern "C" int puts(const char* s);

int main() {
    int n = 5;

    while (n > 0) {
        if (n == 1) {
            [[scpp::unsafe]] {
                puts("Ignition!");
            }
        } else {
            [[scpp::unsafe]] {
                printf("T-minus %d\n", n);
            }
        }
        n = n - 1;
    }

    [[scpp::unsafe]] {
        puts("Liftoff!");
    }
    return 0;
}
```

编译并运行：

```sh
./build/scpp countdown.scpp
./a.out
```

预期输出：

```text
T-minus 5
T-minus 4
T-minus 3
T-minus 2
Ignition!
Liftoff!
```

## 按执行顺序读一遍

这个程序已经让你养成几种核心习惯：

- 从 `main` 开始。程序就是从这里执行的。
- 用变量保存状态。这里的 `n` 记住当前的倒计时数字。
- 需要重复时，用 `while`。
- 某一步是否执行，要看当前值时，用 `if` / `else`。
- 变量要自己更新。`n = n - 1;` 让循环继续向前推进。

虽然这仍然是个玩具程序，但它已经很像真正的代码了：有状态、有分支、有重复、
也有肉眼可见的输出。

## 关于这里的打印调用

我们现在仍然借用了两个 C 库函数 `printf` 和 `puts` 来让例子有可见输出。也正
因为如此，这些调用要放在 `[[scpp::unsafe]]` 代码块里。本章的重点是先让你建
立“一个完整程序长什么样”的感觉，而不是现在就把 FFI 和安全边界完整拆开讲。

## 改写期间暂时保留的参考附录

下面保留的是较早写成、偏参考手册风格的内容。后面的章节目前仍然会链接到这些
材料，所以在教程化改写分批落地期间，它们暂时继续留在这里。

每个函数都**默认、无条件**受检查（见 [§5](ch05-static-checks.md)）——不
存在任何按函数、按文件的标注来开启这件事。语言里唯一的安全上下文构造是
`[[scpp::unsafe]]` 这个 attribute：施加到一个复合语句上时，让这个词法
作用域的块变成一个局部的、可组合的逃生舱，只放宽
[§5.5](ch05-static-checks.md) 里那份固定、列举出来的操作清单；改成施加
到一个函数自己的声明上时，让"调用这个函数"本身也变成这样一项操作，并且
让这个函数的**整个**函数体全程都是 unsafe context（见
[§1.2](#12-标记整个函数嵌套-vs-函数级标记)）。不管哪种情形，
`[[scpp::unsafe]]` 都不会放宽别的任何东西。`scpp::unsafe` 是一个
**attribute**，不是关键字——scpp 压根没有引入任何新关键字（见
[ch00](ch00-design-philosophy.md) 第 2 条）。这是对早先设计（一个
`safe` 关键字，让单个函数 opt-in 检查，没标注的函数完全不检查）的刻意
反转——完整理由见 [§8](ch08-open-questions.md) Q13；这条反转跟下面讲的
函数级标记完全正交、互不影响——不管一个函数有没有标记，它都照样拿到
完整的 [§5.1-§5.4](ch05-static-checks.md) 检查，无条件。

## 1.1 `[[scpp::unsafe]]` 放宽了什么，什么保持无条件

- **[§5.1](ch05-static-checks.md)-[§5.4](ch05-static-checks.md) 里的
  所有东西**——所有权/move 状态、别名 XOR 可变、生命周期/悬垂检查、
  零初始化——**无条件、任何地方都跑，包括在 `[[scpp::unsafe]]` 这个
  上下文里面**。`[[scpp::unsafe]]` 从来不会关掉这套分析；scpp 里也没有
  任何构造能做到这一点。这跟 Rust 完全一致：Rust 的 `unsafe fn` 函数体、
  或者 `unsafe { }` 块内部，借用检查照样完整地跑——`unsafe` 只解锁借用
  检查器自己没法验证的几个具体操作，从来不会把借用检查整个停掉。
- **只有 [§5.5](ch05-static-checks.md) 列出的操作，在
  `[[scpp::unsafe]]` 上下文里才变合法**：裸指针解引用/指针算术、
  `reinterpret_cast`/不兼容类型间的 C 风格强转、未加标签的 `union`
  成员访问、裸 `new`/`delete`、可变全局/静态变量访问、调用一个
  `extern "C"` 函数（`extern "C"` 声明永远不会被任何 scpp 编译器检查，见
  [§2.1](ch02-boundary-rules.md)，所以调用它跟清单上其它每一项
  一样，都需要 `[[scpp::unsafe]]` 提供的这份担保）、调用一个自己
  被标记为 `[[scpp::unsafe]]` 的函数，以及通过一个自身类型就是
  `[[scpp::unsafe]]`-qualified 的函数指针去调用（见
  [§5.16](ch05-static-checks.md#516-函数指针function-pointers)）。
- **`span` 的边界检查（[§8](ch08-open-questions.md) Q1）和整数
  溢出检查（[§5.8](ch05-static-checks.md)）在 `[[scpp::unsafe]]`
  上下文里也会被放宽**，但理由跟上面不一样：不是因为它们在别处不合法（两者
  在哪里都是一直合法的普通语法），而是因为跳过一个 scpp 自己加的
  **运行时**检查，完全没有
  [§5.1](ch05-static-checks.md)-[§5.4](ch05-static-checks.md)
  那种"损坏的记账可能泄漏到外面代码"的风险，所以能保持无条件（完整
  理由见 [§5.8](ch05-static-checks.md)）。

## 1.2 标记整个函数：嵌套 vs. 函数级标记

想让整个函数体大范围访问 [§5.5](ch05-static-checks.md) 的操作，有两种
写法，回答的是两个不同的问题。

**嵌套一个块**回答的是"这个函数**自己的实现**需不需要大范围访问"：
把**整个**函数体包进一个 `[[scpp::unsafe]] { }` 块，在里面嵌套一层。
这不会改变谁能**调用**这个函数——普通、没标记的代码照样能像调用任何
别的受检查函数一样自由调用它：

```cpp
int legacy_style_function(int* p, size_t n) {
    [[scpp::unsafe]] {
        // 整个函数体都在这里；§5.1-§5.4 照样全程生效
    }
}
```

**函数级标记**回答的是另一个问题："调用这个函数本身，要不要求调用者
处在 unsafe 上下文里"。把 `[[scpp::unsafe]]` 直接挂在函数自己的声明上
（放在返回类型前面那个位置——真实 C++ 里 `[[noreturn]]` 用的就是这个
位置），两件事会同时发生（见 [§1.3](#13-scppunsafe-attribute)）：
函数的整个函数体变成 unsafe context，效果跟上面嵌套一层块一样——**而且**，
调用这个函数这件事本身，也变成 [§5.5](ch05-static-checks.md) 里的一项
受管制操作，普通、没标记的调用者不再能直接调用它，必须把调用包进
`[[scpp::unsafe]] { }` 里：

```cpp
// 调用者必须自己保证 index 没有越界；函数体内部没有任何办法验证这一点。
[[scpp::unsafe]] int get_unchecked(int* base, int index) {
    return base[index];   // 这里不需要再嵌套一层 [[scpp::unsafe]]：
                           // 因为上面声明上的这个 attribute，整个函数体
                           // 本身就已经是 unsafe context 了
}

int caller(int* arr) {
    [[scpp::unsafe]] {
        return get_unchecked(arr, 2);   // 担保的是这次调用本身，
                                        // 不只是某个裸指针解引用
    }
}
```

这跟 Rust 的 `unsafe fn` 是同一回事：一个普通函数，哪怕内部包含一个
`unsafe { }` 块，也照样能被任何地方调用，因为它的作者已经针对**这个
函数能接受的所有输入**，对那个具体的内部操作做过担保了；`unsafe fn`
存在的意义正相反——函数的健全性依赖一个**调用者**必须保证的前置条件
（比如"`index` 没有越界"），这个前置条件不管函数体内部写得多小心都没法
自己验证——所以 Rust 强迫这份责任往外传播，传到每一个调用点，而不是
让它藏在一个看不见的实现细节里。scpp 的函数级 `[[scpp::unsafe]]` 是
完全相同的机制，只是拼写成了 attribute 而不是关键字。函数的健全性依赖
只有调用者才能保证的东西时，就用它；函数自己的实现需要大范围访问、但
不管什么输入都能自己担保得住、希望普通代码还能自由调用它时，就用普通的
嵌套块。

这两种写法都不会改变 [§5.1-§5.4](ch05-static-checks.md) 的检查：不管
函数体是被哪种写法（或者都没有）变成 unsafe context 的，所有权/move/
别名/生命周期检查照样对函数体内每一条语句生效，跟
[§1.1](#11-scppunsafe-放宽了什么什么保持无条件) 说的完全一样。这里
特别要注意：scpp **没有**照搬 Rust 自己在 2024 edition 收紧的
`unsafe_op_in_unsafe_fn`——那条规则额外要求一个 `unsafe fn` 自己的函数
体内部，把用到的 unsafe 操作再显式套一层内层 `unsafe { }`；那条规则
提升的是大函数内部的审计清晰度，并不改变任何健全性保证（一个
`unsafe fn` 反正已经在每个调用点被完整把关了），所以 scpp 不照搬它——
函数级 `[[scpp::unsafe]]` 直接让整个函数体变成 unsafe context，不需要
再套一层。

## 1.3 `[[scpp::unsafe]]` Attribute

- **语法**：`scpp` attribute 命名空间下的 attribute-token `unsafe`，可以
  施加在两个位置之一：
  - 施加到一个普通花括号复合语句自己的 attribute-specifier-seq 上：
    `[[scpp::unsafe]] { stmt; stmt; ... }`；或者
  - 施加到一个函数自己的声明或定义、放在返回类型前面那个最前面的
    attribute-specifier-seq 位置上：`[[scpp::unsafe]] int f(int* p) {
    ... }`（如果一个函数被声明了不止一次，每一处声明都必须一致地
    重复这个 attribute——形式化规范里
    [§5.1](../../spec/zh/01-unsafe.md#51-attribute属性dclattrscppunsafe)
    的 (2) 有这条规则）。

  这**完全没有引入新语法**：真实 C++ 本来就给每一个复合语句配了一个
  可选的、前置的 attribute-specifier-seq（`[[likely]] { ... }` 用的就是
  同一个语法位置），也本来就给每一个函数声明/定义配了一个可选的、最前面
  的 attribute-specifier-seq（`[[noreturn]]` 用的就是同一个语法位置），
  所以 `scpp::unsafe` 只是这两个既有产生式里的一个新 attribute-token，
  不是新关键字，也不是新的语法产生式——跟这本书里"零新关键字"这条原则
  下的其它构造不一样，这里根本没有什么是真实 C++ 编译器解析不过去的，
  只有一个它不认识的 attribute（会被悄悄忽略，或者可以用类似
  `-Wno-unknown-attributes` 这样的选项让它连警告都不报）。v0.1 只支持
  这两种位置。Circle/Safe C++ 还允许把 `unsafe` 直接标在单个条件表达式、
  match 分支体、构造函数的 mem-initializer 前面；scpp 因为还没有
  match 表达式、也没有构造函数，这些都先放着——以后各自的位置落地时，
  可以单独加成新的 attribute 用法，不影响这里定的规则。
  - **唯一不好使的位置**：紧跟在函数参数列表后面的
    attribute-specifier-seq——比如 `int f(int* p) [[scpp::unsafe]] { ... }`
    ——挂的是函数的**类型**（[dcl.fct]），既不是函数本身、也不是函数体，
    所以完全没有效果；标记整个函数体的两种正确写法见上面
    [§1.2](#12-标记整个函数嵌套-vs-函数级标记)。
- **作用域**：跟 Circle 不一样（Circle 里 `unsafe { }` 故意**不**开新的
  词法作用域），scpp 的 `[[scpp::unsafe]] { }` 是一个**普通的块**：跟
  普通的 `{ }` 复合语句完全一样——里面声明的局部变量在 `}` 处结束生命
  周期，跟普通 C++ 一样。如果某个 `{ }` 悄悄跟语言里其他所有 `{ }`
  作用域规则不一样，就违反了 scpp 自己"看起来像 C++、不搞意外"这条
  原则。如果 `[[scpp::unsafe]] { }` 里算出来的值需要在外面继续用，按
  普通 C++ 的写法，先在外层作用域声明，再在里面赋值即可。（函数级标记
  同样不引入任何新作用域——函数体本身的作用域不受影响。）
- **解锁了什么**：[§5.5](ch05-static-checks.md) 列的那些操作，在
  `[[scpp::unsafe]]` 上下文里恰好被解锁，且**仅**解锁这些：裸指针解引用/
  指针算术、`reinterpret_cast`/不兼容类型间的 C 风格强转、未加标签的
  `union` 成员访问、裸 `new`/`delete`、可变全局/静态变量访问、调用
  一个 `extern "C"` 函数、调用一个自己被标记为 `[[scpp::unsafe]]`
  的函数，以及通过一个自身类型就是 `[[scpp::unsafe]]`-qualified 的
  函数指针去调用（见
  [§5.16](ch05-static-checks.md#516-函数指针function-pointers)）。
- **`[[scpp::unsafe]]` 上下文里还额外放宽了这些，但理由不一样**：`span`
  的边界检查（[§8](ch08-open-questions.md) Q1）和整数溢出检查
  （[§5.8](ch05-static-checks.md)）也会被跳过——但跟上面那些不一样，
  这两个都不是"在 unsafe 之外就不合法"的操作（在哪里都是一直合法的
  语法）。它们是 scpp 自己加的**运行时**检查，碰巧完全没有下一条讲的
  "泄漏到外面代码"那种风险，这就是它们能加进这份放宽清单、又不
  跟下一条矛盾的原因。
- **什么保持一样严格**：[§5.1](ch05-static-checks.md)-
  [§5.4](ch05-static-checks.md) 里的所有东西——所有权/move 状态、别名
  XOR 可变、生命周期/悬垂检查、零初始化——在 `[[scpp::unsafe]]` 上下文的
  语句里**无条件**继续跑，不管这个上下文是靠哪种写法建立起来的，没有
  例外。`[[scpp::unsafe]]` 是一个针对具体操作的窄口子，**不是**"这段
  区域别检查了"的总开关，不能等价于"跳过对这段语句的所有权/借用/生命
  周期检查"，只能靠放宽上面列的那几条具体检查。不管代码在不在
  `[[scpp::unsafe]]` 里面，`std::unique_ptr` 都不能被 double-move、
  别名 XOR 可变都不能被违反——否则紧跟在块后面（仍然处在完整检查下）
  的代码就可能看到一个从来没被证明过安全的、已经损坏的状态。（裸指针
  本身不管在不在 `[[scpp::unsafe]]` 里都**依然不做** move/borrow
  跟踪——这个不变，跟"裸指针本来就不是被跟踪的类型"这条现有规则是一回
  事。）
- **嵌套**：`[[scpp::unsafe]] { }` 可以嵌套在另一个 `[[scpp::unsafe]]`
  上下文（不管是哪种写法建立的）里——是无害的，不是错误。
