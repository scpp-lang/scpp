# 0. 设计理念（不可动摇的北极星）

1. **视觉上就是 C++**：任何熟悉现代 C++ 的人第一眼就认为这是 C++。
2. **加法极简，而且要能干净地擦除回去**：只在绝对必要时引入新语法，而且
   前提是删掉它之后剩下的还是一个真正的 C++ 编译器能原样接受的普通文件。
   核心新增只有 `unsafe`：把这个关键字删掉（保留花括号里的内容），
   剩下的就是标准 C++——**scpp = C++ + 一个检查器**，不是一门有自己语法的
   新语言。这也是为什么生命周期分组复用 attribute 语法而不是发明 `'a` 那种
   新 token（见 [§5.3](ch05-static-checks.md)），也是为什么 scpp 最终放弃了
   类似 Rust `?` 的错误传播运算符、改用普通 `if`/`else`（见
   [§5.6](ch05-static-checks.md)/[§8](ch08-open-questions.md) Q8）：真正的
   C++ 编译器可以悄悄无视一个不认识的 attribute，但绝对没法越过一个全新的
   运算符 token 继续解析下去。
3. **复用已知语法、重新赋予语义**：`std::move()`、`T&`、`unique_ptr`、`span`
   等既有写法，被赋予更强的静态含义（所有权/借用/生命周期），
   但不改变它们对用户的"外观"。
4. **安全是默认状态；`unsafe` 是唯一的退出方式，而且是局部的、可组合
   的**：每个函数都无条件受检查（见 [§5](ch05-static-checks.md)），不需要
   任何按函数/按文件的标注来开启。`unsafe { }` 是一个词法作用域块，可以
   嵌套、局部可逆，只放开 [§5.5](ch05-static-checks.md) 里那份固定、列举
   出来的操作清单——它从来不是一个能把整个函数或整个文件检查关掉的开关。
   这是对早先设计（一个 `safe` 关键字标注哪些函数 opt-in 检查，没标的
   函数完全不检查）的刻意反转——理由见 [§8](ch08-open-questions.md) Q13。
5. **健全性优先于兼容性**：宁可暂时"尚未支持"某语法，也不
   放行一个不健全的检查。100% C++ 兼容不是目标。
6. **可擦除性压倒人体工程学和功能完整性**：目前没有别的给 C 系语言补
   Rust 级别健全性的尝试，能在保留第 2 条这条可擦除性质的**同时**还真
   做到——保持可擦除的（比如微软的 SAL 标注）力度都远不如 Rust；真做到
   健全的（Sean Baxter 的 Circle/Safe C++）需要自己的编译器，彻底放弃了
   可擦除性；处在中间地带的（Checked C）恰恰在需要新语法表达的那些构造
   （指针类型、边界标注）上放弃了可擦除。**两头都要**，是 scpp 真正的
   赌注和差异化所在。所以以后任何一个新特性，如果非要引入全新的、不可
   擦除的语法才能完整表达、或者才能做到 Rust 那种人体工程学，答案**不是**
   加这个语法：想办法用已经能干净擦除的写法（哪怕啰嗦一点）表达同样的
   东西，或者干脆先不支持这个特性，等想到办法再说（`??` 运算符就是这条
   原则第一次具体落地的例子，见 [§8](ch08-open-questions.md) Q8）。啰嗦、
   功能暂缺，都是能接受、能挽回的代价；可擦除性被打破，不能接受。
7. **正式名字：SCPP26。参照的 C++ 标准：C++26。** scpp 的正式版本化名字，
   直接把它跟参照的标准绑在一起。总目标：**默认支持 C++26 的全部语法**，
   只排除一小撮明确列举出来的、真实 C++ 自己已经承认有缺陷的情况（有据
   可查的坑、明确被 deprecate 的写法，或者类似情况）——绝不是因为 scpp
   自己的设计更偏好某种更显式/不一样的写法（参见
   [§5.12](ch05-static-checks.md) 的 `this` 必须显式 capture 这条规则，
   建立在真实 C++20 自己的 P0806R2 deprecation 之上，是本文档里其它所有
   限制都应该照着走的范本）。另外，本文档里凡是说某个名字/特性是
   "真实的"/"已经标准化"的 C++（比如 `<cstdint>`、`<stdfloat>`、
   `std::expected`、C++20 modules）、还是"还不是标准"（比如
   `int128_t`/WG21 P1467——见 [§6](ch06-safe-subset.md)），分界线**就是
   C++26**：2025 年 6 月 Sofia 会议功能冻结，2026 年 3 月 London 会议
   完成最终投票。第 2 条的可擦除性和第 6 条"只用已批准的名字"这条规则，
   检验基准都是这一版。以后新标准批准了，这条基准要跟着往前挪。
8. **任何编译器自己能控制的地方，绝不允许 UB——哪怕在 `unsafe` 里也不
   行。** 开发者必须永远能知道自己代码的确切行为。真实 C++ 里，哪怕程序
   什么"解引用一个乱指针"这种过分的事都没干，某些操作也是未定义的——
   有符号整数溢出就是最典型的例子（见 [§5.8](ch05-static-checks.md)）：
   编译器有权假设它永远不会发生，然后把（不知情地）依赖了这个假设的代码
   编译错——纯粹是因为标准这么写了，不是因为这件事没法定义清楚。scpp
   绝不给自己这张牌：只要一个操作的"是否有定义"是 scpp 自己 codegen 能
   选择的（整数溢出是第一个具体案例），要么无条件继续检查（默认状态
   下），要么在 `unsafe` 里把它钉死成一个具体的、确定的、写清楚的
   结果（比如保证的补码 wrap）——绝不是真正的 UB。这条不适用于（也没法
   消除）另一种 UB——来自**外部、没法验证的前置条件**那一种（解引用一个
   裸指针，只有指针真的有效才是健全的，这个没有任何静态分析能在一般情况
   下证明）——这一类上，`unsafe` 的含义跟 Rust 里完全一样。
