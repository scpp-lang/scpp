# 1. 安全上下文（Safety Context）

编译器为每个函数体、lambda、块、类型维护一个 **safety context**，取值：

- **`unsafe`（默认，本书里也叫"native 函数"）**：普通 C++ 语义。不做任何
  所有权/借用/别名检查，也没有 scpp 加的运行时检查。
- **`safe`**：启用全部静态安全检查（见 [§5](ch05-static-checks.md)），
  加上 scpp 自己的运行时检查（`span` 边界、整数溢出——见
  [§5.5](ch05-static-checks.md)/[§5.8](ch05-static-checks.md)）。

## 1.1 上下文的确定规则

- 未标注的函数/块 → `unsafe`（默认）——本书里管这个叫 **native 函数**：
  普通 C++，没有任何 scpp 检查，连
  [§5.1-§5.4](ch05-static-checks.md) 那些 compile-time 检查都没有。这是
  描述性的说法，不是关键字——没有 `native` 这个 token 要写，单纯不写
  `safe` 就落在这个状态。
- `safe` 标注的函数/块 → `safe`。
- 子块继承父块的上下文，除非被显式标注覆盖。
- 写在 `safe` 函数**里面**的 `unsafe { }` 块，会开一个逃生窗口，局部放宽
  [§1.3](#13-unsafe--块设计已定稿尚未实现) 里列的那些东西。
- `safe { }`/`unsafe { }` 出现在 **native** 函数里，两个都是编译错误，
  这是永久的设计决定，不是暂时的缺口：native 函数本来就没有任何生效的
  检查可以放宽或者重新开启——见 [§1.3](#13-unsafe--块设计已定稿尚未实现)
  的"嵌套"部分。

## 1.2 标注位置

```cpp
safe int f(...);                 // 自由函数
struct S { safe void g(); };     // 成员函数
safe struct Widget { ... };      // 类型级：所有成员函数默认 safe（v0.2 再定）
auto lam = safe [](int x){...};  // lambda（v0.2 再定）
```

`safe` 是**前置**标注，写在返回类型前面——不是像 `noexcept`/`override`
那样的后置修饰符。这一点重新考虑过（一度改成后置、模仿 Sean Baxter 的
Circle/Safe C++ 那种 `noexcept`-like 位置），但 `safe` 在 scpp 里注定要
成为一个代码库里**主流、被鼓励达到的常态**，而不是像普通 C++ 里
`override`/`noexcept` 那样偶尔才用一次的例外。前置关键字是最便于扫描的
位置——不管参数列表多长、要不要换行，它永远出现在同一个可预期的地方，
读者可以直接沿着文件左边缘往下扫，一眼看出哪些函数是 safe 的，跟今天
扫描 `virtual`/`inline`/`explicit`/`constexpr` 是一个道理。
[§5.3](ch05-static-checks.md) 的 `[[scpp::lifetime(name)]]` 不受这个决定
影响，依旧后置——attribute 本来就没有"前置"这个选项：
`safe int& foo(...) [[scpp::lifetime(a)]] { ... }`。

v0.1 **只要求实现函数级 `safe` 和块级 `unsafe { }`**。其余标注位置进入
backlog。

## 1.3 `unsafe { }` 块（设计已定稿，尚未实现）

- **语法**：`unsafe` 后跟一个普通的花括号语句列表，可以出现在函数体内任何
  能写语句的地方：`unsafe { stmt; stmt; ... }`。v0.1 只支持这一种块语句
  形式。Circle/Safe C++ 还允许把 `unsafe` 直接标在单个条件表达式、
  match 分支体、构造函数的 mem-initializer 前面；scpp 因为还没有
  match 表达式、也没有构造函数，这些都先放着——以后各自的语法落地时，
  可以单独加成新的语法产生式，不影响这里定的规则。
- **作用域**：跟 Circle 不一样（Circle 里 `unsafe { }` 故意**不**开新的
  词法作用域），scpp 的 `unsafe { }` 是一个**普通的块**：跟普通的 `{ }`
  复合语句完全一样——里面声明的局部变量在 `}` 处结束生命周期，跟普通
  C++ 一样。如果某个 `{ }` 悄悄跟语言里其他所有 `{ }` 作用域规则不一样，
  就违反了 scpp 自己"看起来像 C++、不搞意外"这条原则。如果 `unsafe { }`
  里算出来的值需要在外面继续用，按普通 C++ 的写法，先在外层作用域声明，
  再在里面赋值即可。
- **解锁了什么**：[§5.5](ch05-static-checks.md) 列的那些操作，在
  `unsafe { }` 里恰好被解锁，且**仅**解锁这些——语义和写在普通
  （未标注/`unsafe`）函数里完全一样：裸指针解引用/指针算术、
  `reinterpret_cast`/不兼容类型间的 C 风格强转、未加标签的 `union`
  成员访问、裸 `new`/`delete`、可变全局/静态变量访问、调用未标注
  `safe` 的函数。这些里**目前只有两条在当前实现里能真正碰到**：裸指针
  `*p` 解引用（`T*` 现在就是一个真实类型），以及调用非 `safe` 函数
  （函数调用和 `Function::is_safe` 都已经存在——不过"被调方必须
  `safe`"这条拒绝规则本身现在也还没实现，所以上线 `unsafe { }` 的同时，
  也得把这条检查一起做出来，用同一套机制去挡）。另外四条现在完全没有
  对应语法（`union`、`reinterpret_cast`、`new`/`delete`、全局变量在
  v0.1 里都还没有词法/语法支持）——这些语法落地之前无从"解锁"；等它们
  各自落地时，接入同一套机制即可。
- **`unsafe { }` 里还额外放宽了这些，但理由不一样**：`span` 的边界检查
  （[§8](ch08-open-questions.md) Q1）和整数溢出检查
  （[§5.8](ch05-static-checks.md)）也会被跳过——但跟上面那些不一样，
  这两个都不是"在 unsafe 之外就不合法"的操作（在哪里都是一直合法的
  语法）。它们是 scpp 自己加的**运行时**检查，碰巧完全没有下一条讲的
  "泄漏到外面 safe 代码"那种风险，这就是它们能加进这份放宽清单、又不
  跟下一条矛盾的原因。
- **什么保持一样严格**：[§5.1](ch05-static-checks.md)-
  [§5.4](ch05-static-checks.md) 里的所有东西——所有权/move 状态、别名
  XOR 可变、生命周期/悬垂检查、零初始化——在 `unsafe { }` 块的语句里
  **无条件**继续跑，没有例外。`unsafe { }` 是一个针对具体操作的窄口子，
  **不是**"这段区域别检查了"的总开关，实现时绝不能靠"跳过对这段语句跑
  movecheck"，只能靠放宽上面列的那几条具体检查。不管代码在不在
  `unsafe { }` 里面，`std::unique_ptr` 都不能被 double-move、别名
  XOR 可变都不能被违反——否则紧跟在块后面（仍然处在完整 `safe` 检查下）
  的代码就可能看到一个从来没被证明过安全的、已经损坏的状态。（裸指针
  本身不管在不在 `unsafe { }` 里都**依然不做** move/borrow 跟踪——这个
  不变，跟"裸指针本来就不是被跟踪的类型"这条现有规则是一回事。）
- **嵌套**：`unsafe { }` 可以嵌套在另一个 `unsafe { }` 里——只要两层都还
  在 `safe` 函数的词法范围内，这样是无害的，不是错误。但 `unsafe { }`
  （或者 `safe { }`）写在一个 **native** 函数的函数体里，**是编译
  错误**：native 函数没有任何生效的检查可以让这两种块去放宽或者重新
  开启，这个标记在那里不只是多余，而是**没有意义**——拒绝它还能顺便抓
  出"代码在 safe 函数和 native 函数之间搬动时忘了清理"这类情况。（这
  收紧了这一节早先的草稿——之前把 `unsafe { }` 出现在 native 函数里
  当成无害的 no-op，而不是错误。）`safe { }` 在 native 函数里重新开启
  检查，同样是一个永久的设计决定，不是暂时的缺口（跟
  [§1.1](#11-上下文的确定规则) 里的说法一致，只是那边换了个说法，理由
  相同）。
- **实现形状**（给做这个的人）：复用现有的 `StmtKind::Block` AST 节点
  （加一个 `is_unsafe` 标记），并且模仿现有的 `ScopeExit` MIR 模式，加一对
  对称的标记语句（比如 `UnsafeEnter`/`UnsafeExit`）把这个块 lower 出来的
  语句括起来。movecheck 的遍历只需要维护一个简单的嵌套计数器
  （`UnsafeEnter` 时 +1，`UnsafeExit` 时 -1），跟现有遍历逻辑并行即可——
  这个计数器**不需要**像 `DataflowState` 那样跨 CFG 分支做 join，因为它
  在每个程序点上是纯词法/结构性的事实，不是流敏感的（`if` 的每个分支
  在走到汇合点之前，自己内部的 `unsafe { }` 块早就已经闭合了）。上面
  提到的两条目前能碰到的检查，都在 `counter == 0` 时才生效；`span` 的
  边界检查和整数溢出检查（等实现的时候）也用同一个计数器，不需要另外
  搞一套机制。拒绝 native 函数里的 `unsafe { }`/`safe { }`，是完全独立
  的、简单得多的检查，跟这个计数器没关系——只需要在解析/消解阶段看一眼
  外层函数自己的 `Function::is_safe`，跟嵌套状态无关。
