## 2026-07-11：scpp 是否应该允许从整数到枚举类型的未检查 C 风格强制转换？

### 背景

在 PR #119 中实现 `scpp::rand::uniform_int_distribution` 的 `make(...)`
工厂时，早期草稿里曾经出现过这样的代码：

```cpp
error code = (error)0;
```

那个具体位置后来已经改成直接使用具名枚举值 `error::empty_range`，这样更清晰，
也避免把“这个枚举值底层整数是多少”这样的知识硬编码进调用点。

但更广泛的语言问题仍然没有结论：当一个整数值可能根本不对应任何具名枚举值时，
scpp 是否应该允许把它直接做成枚举类型的裸数值转换？

### 开放问题

目前至少有两个看起来都说得通的方向：

1. **像 C++ 一样，允许自由使用原始的 C 风格强制转换**：

   ```cpp
   auto e = (SomeEnum)int_value;
   ```

   这样做和 C++ 的预期最兼容，但它也是一个典型的 footgun。转换得到的枚举值
   可能根本不对应任何已声明的枚举项。这也许“只是”一个逻辑 bug，但它也可能
   和后续代码产生很糟糕的交互，例如某个 `switch` 语句默认假定只会出现具名
   case、而没有写 `default` 分支。

2. **改为要求通过一个显式、具名的转换设施**，例如：

   ```cpp
   auto e = scpp::enum_cast<SomeEnum>(int_value);
   ```

   在这个模型里，语言和/或标准库会把转换策略明确表达出来。这个转换可以先检验：
   该整数是否真的对应一个实际存在的枚举项；若不对应，则以某种经过刻意设计的方式
   报告失败，而不是悄悄构造出一个可能无效的枚举值。

   至于“失败”到底该怎么表达，也还没有定论：`std::expected`、类似 optional
   的失败返回、trap/abort、在常量求值语境中直接编译报错，还是别的方案，都还
   是开放问题。

### 当前状态

目前还没有决定 scpp 最终应该：

- 允许未检查的“整数到枚举”C 风格强制转换，
- 对这种转换施加限制，
- 禁止它并强制使用更安全的显式设施，
- 或者拆成“安全默认路径 + 不安全逃生口”两种形式。

等以后有时间更系统地思考枚举不变量、转换易用性、诊断体验以及兼容性目标时，
再回过头来讨论这个问题。
